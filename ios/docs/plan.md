# DVM Finance iOS — Implementation Plan (v1)

Executes [spec.md](spec.md). Phases run in order; each phase is implemented
by a subagent, reviewed against the spec by the orchestrator, and committed
separately on branch `claude/ios-dvm-finance-app-79t49l`. The Python sources
under `src/abn_combined/` are the authoritative reference for every port.

Because the dev container has no Swift toolchain, each phase ships with
XCTest coverage that the user runs on a Mac (`xcodegen generate` then
Cmd-U / `xcodebuild test`). Code must compile against iOS 17 / Swift 5.9+.

## Phase A — Scaffold, package, database layer

**Deliverables**
- `ios/project.yml` (XcodeGen): app target `DVMFinance` (iPhone+iPad,
  iOS 17.0, SwiftUI lifecycle, bundle id `com.dvm.finance`), local package
  dependency on `DVMFinanceKit`, unit-test target for the Kit.
- `ios/DVMFinanceKit/Package.swift`: library target + test target, dependency
  `GRDB.swift` (from: "6.0.0").
- `Database/`: GRDB record structs for the 7 tables (spec § Data model) with
  desktop column names via `CodingKeys`; `AppDatabase` with a
  `DatabaseMigrator` migration `v1` creating all tables + the desktop's
  indexes; convenience test factory (in-memory DB).
- Minimal `DVMFinanceApp.swift` + placeholder tab views so the project builds.
- `ios/README.md`: prerequisites (Xcode 15+, xcodegen via brew), generate/
  build/sideload steps, how to run tests.
- `ios/.gitignore` (xcuserdata, generated .xcodeproj, DerivedData).

**Acceptance**: record types round-trip through an in-memory GRDB DB in
tests; column names match `core/models.py` exactly.

## Phase B — Core logic: normalization, identity, dedup, rule engine

Port, in `Core/`: `utils.py` → `Normalize.swift` + `TransactionHash.swift`;
`dedup.py` → `TransactionID.swift` + `Dedup.swift`; `categorizer.py`
(matching + `applyRules` + `recordRuleChange`; **no** `preview_rule`) →
`Categorizer.swift`. Use CryptoKit (`Insecure.MD5`, `SHA256`).

**Tests**: port the relevant cases from `tests/` (normalization edge cases,
WERO/, id generation incl. PayPal/Wise/SEB branches, in-batch dup handling,
two-pass engine: manual skip, first-match-wins by priority then id,
no-match clears category, tag-only merge de-dup, source-kept-in-sync,
context filters, AND/OR condition folding, invalid-regex-never-matches).

**Acceptance**: ids and hashes byte-identical to Python for a table of
fixture inputs (fixtures generated by running the Python functions, values
hard-coded into the Swift tests with a comment naming the generator).

## Phase C — Snapshot import/export

Port `core/snapshots.py` → `Snapshot/`: `SnapshotCodec` (gzip via zlib
`compression`/`Compression` framework, JSON with explicit key order not
required), `SnapshotImporter` (validate → backup → single-transaction
incoming-wins merge → uuid remap → `snapshot_imports` + `import` change
report), `SnapshotExporter` (+ `MachineID` persisted UUID).

**Fixtures**: generate `fixture-snapshot.json.gz` with the Python exporter
against a small seeded devdata DB; commit under `Tests/Fixtures/`; test
import counts, incoming-wins overwrite of a locally-differing manual
category, remapping of `categorization_source`, re-import idempotency
(reports not duplicated), and export→import round-trip equality.

**Acceptance**: the fixture snapshot imports with the exact counts the
Python importer reports for the same starting DB state.

## Phase D — Statement parsers

Port to `Parsers/`: `parsers/utils.py`, `description.py` (structured
extraction — required for rule parity), `mt940.py`, `csv.py` (ABN),
`paypal.py`, `wise.py`, `seb.py`, and the format-detection entry point used
by the upload API. Skip `xls.py` (backlog: CoreXLSX).

Numeric fidelity: the Python parsers produce **float** amounts and
transaction ids embed `str(float)` (`"-12.3"`, `"5.0"`). Swift parsers parse
amounts as `Double` and the id component uses Swift's shortest-round-trip
`String(describing: Double)`, which agrees with Python `repr(float)` for
monetary values — the checked-in parity fixtures
(`transaction_id_float_amounts`) are the guard. Convert to `Decimal` only
when constructing the DB record, after the id is computed.

**Fixtures**: copy/derive the statement fixtures the Python tests use into
`Tests/Fixtures/`; for each format assert transaction count, first/last
row's every field, and `description_structured` JSON equality with the
Python parser output for the same file.

**Acceptance**: per-format fixture parity tests pass (on Mac); import flow
(parse → dedup → insert → applyRules → report) covered by an end-to-end
Kit test using an in-memory DB and a bundled rules snapshot.

## Phase E — SwiftUI screens + wiring

- `Query/TransactionFilter.swift` + `TransactionQuery` (GRDB) implementing
  spec filter semantics incl. subtree include/exclude and transfer
  exclusion default; `TrendsBuilder` (month × top-level category rollup).
- Views: `TransactionsView` (searchable, date-grouped, filter sheet,
  detail), `TrendsView` (matrix, period picker, transfers toggle,
  cell tap → TransactionsView with prefilled filter), `ImportView`
  (fileImporter for statements incl. UTType registrations for .sta/.940/
  .csv/.json.gz, snapshot import, snapshot export via ShareLink, audit
  history lists), `DVMFinanceApp` TabView wiring, basic empty states.
- Filter/trends logic lives in the Kit with unit tests; views stay thin.

**Acceptance**: Kit tests for filter subtree/exclude/transfer semantics and
trends rollup sums; app compiles with no logic in the app target beyond
view code.

## Phase F — Final review & handoff

Orchestrator: spec-conformance review of the full diff, README polish,
backlog section (XLS, budgets/cash-flow UI, rule editing, manual
categorization as v1.1 candidates), squash-free commit history, push.

## Working agreements for subagents

- Read the named Python source file(s) before writing the Swift port; cite
  the origin in each type's doc comment (`/// Port of core/dedup.py:...`).
- Never invent semantics: when Python behavior is surprising (e.g.
  whitespace-then-WERO ordering, `categorization_source` sync on no-change),
  port the surprise and add a test for it.
- No new dependencies beyond GRDB without orchestrator approval.
- Keep everything inside `ios/`; never touch the Python package.
- Commit messages follow repo convention; never mention AI tooling.
