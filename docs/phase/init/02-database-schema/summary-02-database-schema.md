# Summary — 02 Database Schema

## Completed
2026-07-07

## Goal
All SQLAlchemy 2.x models and the initial Alembic migration exist; the app
creates/updates its SQLite schema automatically on startup.

## What Was Built
- `core/models.py` (SQLAlchemy 2.x `DeclarativeBase` + `Mapped`/`mapped_column`):
  - Ported `Transaction`, `CategorizationRule`, `RuleCondition`, `Budget` with legacy
    column names preserved (`accountNumber`, `transactiondate`, ...).
  - `CategorizationRule.uuid` — new, unique, `default=uuid4`, indexed.
  - New tables: `rule_change_reports` + `rule_change_items` (cascade), `download_state`
    (unique source+account), `snapshot_imports`. JSON columns for before/after/summary/
    counts/overwrites. `RULE_CHANGE_ACTIONS` tuple documents the action vocabulary.
  - Dropped users/auth models entirely.
- Alembic: `alembic.ini`, `alembic/env.py` (targets `Base.metadata`, resolves DB URL from
  programmatic override or the data-dir DB), `script.py.mako`, and one autogenerated
  initial revision creating all 8 tables + indexes.
- `migrations.upgrade_to_head` runs `alembic upgrade head` on app startup (create_all
  fallback retained only for non-editable installs).
- Session management already in `db.py` (`configure_engine`, `get_db`).
- Tests: `test_models.py` (round-trips per table, uuid auto-gen + uniqueness, rule/condition
  and report/item cascade deletes, defaults), `test_migrations.py` (fresh startup creates all
  tables, `alembic current` == head, second startup no-op). +11 tests (26 total).

## Key Decisions
- JSON via SQLAlchemy `JSON` type (SQLite stores as TEXT) for report/snapshot payloads.
- `render_as_batch=True` in alembic env for SQLite-friendly future migrations.

## Deviations
- The startup runner locates `alembic.ini`/`alembic/` relative to the repo root, which works
  for editable installs; the create_all fallback covers packaged (uvx) installs until the
  alembic tree is bundled into the wheel (revisit in step 13/packaging).
